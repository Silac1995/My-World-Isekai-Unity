Variant A · Radios + Sell checkbox

Pro: Tool/Inventory cardinality is visually obvious — radios = pick one, checkboxes = pick many.

Con: Inventory column hard-codes "multi". If Inventory is ever single, this needs swapping to radios — schema-coupled.

Open Q1 answer: Reads OK at 5 rows; gets crowded at 10+ with 3 columns of widgets right of slots.

Open Q4 answer: Slot count + bar; no item icons (keeps row height tight).

Variant B · Dropdown per row

Pro: Single role per furniture is conceptually cleanest. Adding a new role = one more dropdown option, no extra column.

Con: Loses cell-grid's at-a-glance "who has what". Sell-Shelf still needs its own column (multi).

Open Q2: Hardest variant to surface inspector-authored fallback inline — would need an italic "(default)" dropdown option.

Variant C · Cell grid

Pro: Pure tabular — most scannable at 10+ storages. Cardinality (single vs multi) encoded by widget shape (radio vs checkbox).

Pro: Adding a role = one more column. Schema changes (Inventory single↔multi) = swap radio↔checkbox in one column.

Con: Most "spreadsheet-y". Icons in header help but designer polish matters.

Recommendation: Best fit for the LogisticsManager mental model + future-proof for new roles.

Cross-variant decisions

Q3 — Inline vs modal: All three are inline — modal-per-furniture loses comparison. Stick with inline.

Q4 — Contents preview: All three show slot count + bar only. Icons-in-slots can land later as a hover preview (no UI cost when collapsed).

Inspector fallback: A and C can show it as a yellow hint banner under the list (currently shown on the Shop context). B would need an italic "(default)" entry in the dropdown.

Terrain & Weather Architecture: Two Approaches

Both use the Hybrid Zone+Cell model you approved. The difference is how the systems are structured.

A

Layered Architecture (Bottom-Up)

Each concern is a standalone system. They communicate via events — minimal coupling.

Layer 5 — Effects
  FootstepAudioResolver · CharacterTerrainEffects (speed/damage/slip)
↑ reads terrain state
Layer 4 — Growth
  VegetationGrowthSystem (wild) · CropSystem (plowed)
↑ reads moisture + fertility
Layer 3 — Weather ↔ Terrain
  TerrainWeatherProcessor (WeatherFront passes over cells → updates moisture, temp)
↑ WeatherFront position + cell grid
Layer 2 — Weather Movement
  WeatherFront (GameObject) · GlobalWindController · BiomeRegion (bounds + spawn rules)
↑ driven by global wind + local wind
Layer 1 — Terrain Foundation
  TerrainType (enum/SO) · TerrainCellGrid · TerrainZone (base type)

Pros

  • Each layer can be built and tested independently
  • Easy to add new terrain types or weather types without touching other layers
  • MacroSimulator only needs Layer 1 + math formulas — no GameObjects
  • Follows your existing SOLID patterns
  • Footstep audio is just a consumer — completely decoupled from weather

Cons

  • More files/classes upfront
  • Event wiring between layers needs careful setup

Recommended — Matches your project's architecture philosophy. Each system has one job. Easy to implement incrementally (Layer 1 first, then 2, etc.)

B

Unified Manager (Top-Down)

A single TerrainWeatherManager orchestrates everything — terrain, weather, growth, effects.

TerrainWeatherManager
  ├── manages TerrainCellGrid
  ├── spawns/moves WeatherFronts
  ├── runs growth ticks
  ├── applies character effects
  └── handles footstep lookups

Pros

  • Fewer files — everything in one place
  • Simple to understand initially

Cons

  • God-class risk — one manager doing 5 jobs
  • Violates SRP from your project rules
  • Hard to test individual concerns
  • Adding new terrain types or effects means modifying the manager
  • Offline simulation tightly coupled to runtime logic

Footstep Audio Matrix

Ground Type × Foot Type → Sound
Bare Feet Leather Boots Metal Boots Hooves
Grass/Fertile soft rustle muffled thud crunching soft thump
Dirt/Mud squelch wet thud heavy squelch splat
Stone light tap click clang clip-clop
Snow crunch deep crunch heavy crunch powdery crunch
Ice scrape scratch metallic scrape sharp click
Wood Floor soft pad creak heavy thud hollow clop

Each cell = an AudioClip array (variations for randomization). Foot type comes from CharacterEquipment boot slot.
